# 技术规范

## 1. 技术栈

### 前端
| 技术 | 版本 | 用途 |
|------|------|------|
| React | ^19.x | UI 框架 |
| Vite | ^6.x | 构建工具 |
| React Router DOM | ^7.x | 路由管理 |
| react-markdown | ^9.x | Markdown 渲染 |

| 技术 | 版本 | 用途 |
|------|------|------|
| Node.js | >= 18 | 运行时 |
| Express | ^4.21.x | Web 框架 |
| cors | ^2.8.x | 跨域支持 |
| better-sqlite3 | ^11.x | SQLite 数据库驱动 |

| 类型 | 存储方式 | 内容 |
|------|---------|------|

| 动态数据 | SQLite 数据库 (`data/anatomy.db`) | 进度、错题、设置 |

## 3. 编码规范

### 3.1 JavaScript / JSX
- 使用 ES Modules (`import` / `export`)
- 函数组件 + Hooks（不使用 Class 组件）
- 组件文件名：PascalCase（`HomePage.jsx`）
- 工具/服务文件名：camelCase（`api.js`, `quizService.js`）
- 不写多余的注释，只在非显而易见的逻辑处加一行注释
- 不在代码中写 TODO/FIXME（用 dev-logs 追踪）

### 3.2 CSS
- 使用原生 CSS（不用 CSS-in-JS 或预处理器）
- CSS 变量定义在 `globals.css` 的 `:root` 中
- BEM 风格命名：`.block__element--modifier`
- 移动端优先，`responsive.css` 中用 `min-width` 向上增强

### 3.3 API 规范
- 路径前缀：`/api/v1`
- 成功响应：
```json
{ "success": true, "data": {...}, "timestamp": "ISO字符串" }
```
- 错误响应：
```json
{ "success": false, "error": { "code": "ERROR_CODE", "message": "...", "details": {} }, "timestamp": "..." }
```

### 3.4 ID 命名规范
| 实体 | 格式 | 示例 |
|------|------|------|
| 模块(章) | `chapter-NN` | `chapter-01` |
| 节 | `section-NN-NN` | `section-01-01` |
| 小节 | `subsection-NN-NN-NN` | `sub-01-01-01` |
| 部分(最小单元) | `part-XXXX` | `part-概述` |
| 测验题 | `qz-NNN` | `qz-001` |
| 测试题 | `tst-NNN` | `tst-003` |
| 真题 | `fe-NNN` | `fe-002` |

## 5. Git 规范

- 每次完成一个**完整的功能点**后再提交
- commit message 格式：`type: 简短描述`
  - `feat:` 新功能
  - `fix:` 修复 bug
  - `refactor:` 重构
  - `docs:` 文档变更
  - `chore:` 构建/工具变更
- 不提交 node_modules、data/*.db、.env 等文件

## 6. 分支策略

- `main`: 稳定版本
- 功能开发直接在 main 上进行（单人项目）
- 每次完成一个实施步骤后确认可运行再继续下一步
